# CLAUDE.md

## Project Overview

K-8 vocabulary learning web app with AI-generated instructional content, spaced repetition practice, and role-based access (admin, teacher, student). Target audience: ages 8–14, ESL learners.

## Tech Stack

- **Backend**: Django 5.2 + DRF 3.16, Python, MySQL
- **Frontend**: React 19 + Vite 7, plain CSS with theme system
- **LLMs**: Gemini (text generation + graphic novel scripts), OpenAI GPT-Image-2 (images), Qwen3 (embeddings via SiliconFlow). Anthropic SDK is wired for fallback but no active step currently uses it.
- **Auth**: Session-based with CSRF tokens

## Configuration

Backend requires `.env` file (see `.env.example`):
- `DATABASE_URL` — MySQL connection string
- `GEMINI_API_KEY` — API key for Gemini (used by `call_gemini`; also used as the auth key when routing through `GEMINI_BASE_URL`)
- `GEMINI_BASE_URL` — Optional. If set, `call_gemini` routes through an **OpenAI-compatible proxy** (e.g., `https://api.b.ai/v1`). The SDK appends `/chat/completions` automatically, so the value must be the API root without that suffix.
- `GEMINI_TTS_API_KEY` / `GEMINI_TTS_BASE_URL` / `GEMINI_TTS_MODEL` — Read-along audiobook TTS, separate from the text Gemini config because audio needs the **native** generateContent API (an OpenAI-compatible text proxy does not serve it). `GEMINI_TTS_API_KEY` falls back to `GEMINI_API_KEY`; leave `GEMINI_TTS_BASE_URL` empty to call Google directly or point it at a proxy that serves native Gemini TTS; `GEMINI_TTS_MODEL` defaults to `gemini-2.5-pro-preview-tts`.
- `ANTHROPIC_API_KEY` — Used when any pipeline step is configured to route through an Anthropic site
- `ANTHROPIC_BASE_URL` — Optional proxy for Anthropic API
- `OPENAI_API_KEY` — GPT-Image-2 for graphic novel images
- `OPENAI_BASE_URL` — Optional proxy for OpenAI API
- `QWEN_API_KEY` — Qwen3 embeddings via SiliconFlow
- `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`

Note: Which API keys are actually needed depends on which sites are configured in the LLM Configuration admin page. The `LLMSite` model references env var names; the backend resolves them at runtime.

MySQL driver: `pymysql` (pure Python, installed as MySQLdb via `config/__init__.py`). No C compiler needed.
`requirements.txt` lists `pymysql` + `anthropic` + `google-genai` + `lameenc` (WAV→MP3 audiobook encoding, pure binary wheel — no ffmpeg) — do NOT add `mysqlclient` (requires C compiler, not used).

Frontend dev server proxies `/api` and `/media` to `localhost:8001`.

## Key Architecture Decisions

- Spaced repetition uses response-quality-aware scheduling (fast/solid/slow affects intervals)
- Words transition PENDING → READY after instructional pack completion; only READY words enter SRS
- Mastery levels 6-7 are hidden from students (rolled into "Mastered" on dashboard)
- Generation pipeline is a modular package at `vocabulary/services/generation/`; `generation_pipeline_service.py` is a backwards-compatible re-export shim
- 11 steps with per-step resume. `_reconstruct_context` rebuilds `words_data` from the WORD_LOOKUP log's `word_lookup_snapshot` when dedup hasn't completed (prevents word loss on mid-dedup failures)
- `GraphicNovel` FK→WordPack. **Multi-candidate**: each pack generates `GRAPHIC_NOVEL_CANDIDATE_COUNT = 3` (constants.py) independent candidate novels (`candidate_index` 0-2), each from its own full team-selection→script workflow, so they diverge in story/art/framing — countering per-call LLM variance. `unique_together = ('pack', 'candidate_index')`. `channel` is now a dead column (always `'5page'`, kept pending removal). `is_selected` marks the one admin-chosen candidate; **no candidate is auto-selected**, so until an admin picks one the pack stays hidden from students (deliberate manual publish gate). `novel.metadata['page_count']` stores the length. Student-facing code (`instructional_service.py`) reads `graphic_novels.filter(is_selected=True)` and iterates saved `GraphicNovelPage` rows (length-agnostic). Migration `0035` backfills existing novels → `candidate_index=0, is_selected=True`
- Candidate selection: `POST /api/graphic-novels/{novel_id}/select/` (admin, `SelectGraphicNovelCandidateView` → `services/graphic_novel_selection_service.select_graphic_novel_candidate`) sets `is_selected` on the chosen novel, clears siblings, and **promotes** that novel's staged cloze to the pack. Idempotent + reversible. Losing candidates are kept (hidden). Audiobook stays manual — admin only triggers `generate-audio` on the selected novel. UI: `GenerationReview.jsx` lays candidates out as a **compare strip + detail** (`PackGraphicNovels` → `CandidateStripRow` thumbnails per candidate + a full-width `CandidateDetail` panel for the focused one); the two admin review content endpoints return `graphic_novels` (array, all candidates each with its own cloze) plus `graphic_novel` (the selected one, or null). The `GenerationJobStatus.jsx` progress views are candidate-grouped too (substeps nested under per-candidate headers, image pages as one chip-row per candidate); both rely on backend `candidate_index` in the page-status + substep-status payloads. Teacher portal shell widened to 1760px (`teacher.css` `.t-shell`)
- Cloze is **per-candidate**: generated inside each candidate's script flow and stored as `ClozeItem(novel=<id>, pack=<pack>)` (staged). Selecting a candidate promotes its rows to `ClozeItem(novel=None, pack=<pack>)` — the active set students practice (prior promoted rows deleted first). `ClozeItem.novel` FK (nullable, migration `0035`): `novel=None` = promoted/active; `novel=<id>` = staged. Student + practice reads filter `novel__isnull=True`
- **Infographic content type** (alternative to graphic novels, migration `0036`/`0037`): teachers who don't want the multi-page comic can assign a single-page **infographic** instead. `Infographic` FK→WordPack mirrors `GraphicNovel` (same 3-candidate + admin-select model, `unique_together=('pack','candidate_index')`, `is_selected`, single image with `display_image`/`student_image` PNG+JPEG variants). Neutral educational style — **no Lexi Legends canon/characters**. **Style target: modern editorial infographic (NotebookLM / data-viz explainer)** — ONE cohesive designed piece that teaches vocab in context, explicitly NOT a flashcard grid AND NOT a children's storybook/picture-book illustration (the v1 prompt produced 6 word→def cards on a backdrop; redesigned 2026-06-24, storybook feel toned down 2026-06-26). Captions USE the word in a sentence (never `word: definition` — validator rejects the glossary format) and are sequential narrative beats, not isolated facts. **Two layout modes** (`layout_mode`, the design LLM acts as art director and picks one): `panorama` = one continuous landscape/map with a visual SPINE connecting words as landmarks (for sequential packs); `gallery` = several vignettes held by a creative FRAMING DEVICE — scrapbook/corkboard/shelf, never a plain grid (for disjointed/abstract packs). `build_infographic_image_prompt` branches the image template's `{layout_guidance}` on `layout_mode` (spine vs framing-device language). **Target-word emphasis on the poster**: `_mark_vocab_terms` wraps each caption's target words in `**markers**` (case-insensitive, word-boundary, suffix-tolerant for plurals/`-ed`/`-ing`, no double-wrap — replaced the old case-sensitive `str.replace`), and the image template + an explicit `{vocab_words}` list (`_all_target_terms`) tell the model to render each marked word in a bolder weight + distinct accent color (asterisks themselves are never printed) so the taught word stands out in its sentence. `ig_design` emits `title` (a single-line big-idea title — no "Main Idea: Subtitle" subtitle; `_clean_infographic_title` strips a trailing generic tag like "A Vocabulary Guide" on persist), `intro_text` (the short hook students read above the poster — must **use every target word in context**; validator rejects any omitted word, stem-tolerant for plurals/`-ed`/`-ing`), `big_idea`/`layout_mode`/`visual_structure`/`scene_description` (names the spine/framing-device + eye-path) / `scene_elements[]` (label/caption/vocab_terms/illustration-with-callout-line) + per-word `entries[]` (study panel + cloze only, NOT on poster). **Art-directed per-candidate style**: `ig_design` also emits `color_palette` (3–5 specific vibrant colors, varied per poster) and `style_prompt` (the rendering look). The design prompt has the LLM act as art director and **mix-and-match** a *distinct* combination of modern editorial-infographic terms from three menus (CORE STYLE: modern flat-vector illustration / clean diagrammatic design / data-visualization explainer look / flat design with soft shading; COLORS & LIGHTING: vibrant cohesive palette / soft gradients / confident editorial color blocking; DETAILS: clean crisp outlines / simplified iconographic figures / labeled callouts + connector lines / banners-ribbons / floating text boxes) per candidate, so the 3 candidates (separate design calls) diverge in look rather than sharing one style. The look is explicitly NOT a children's storybook/picture-book/cartoon illustration (toned down 2026-06-26). `style_prompt` flows `design_result → Infographic.style_prompt → {style_prompt}` in `infographic_image.txt`, where it leads as the art direction; the template's `STYLE_LOCK` is only a thin shared *floor* (modern flat-vector editorial infographic, geometric sans-serif lettering, legible, one unified piece, no photorealism / *photorealistic* 3D render / muddy colors / storybook-or-cartoon look — soft dimensional shading IS allowed) that must not override the per-poster style. `DEFAULT_INFOGRAPHIC_STYLE` in `step_infographic.py` is the fallback when a (e.g. older) candidate lacks `style_prompt`; `style_prompt` is optional in the validator. **Both prompts de-emphasize people** (`infographic_design.txt` + `infographic_image.txt`): kids fixate on faces and the image model distorts figures, so the design favors objects/tools/places/animals/landscapes and the image prompt keeps human figures few, small, and simple (no large close-up faces) — soft guidance, not a ban. Validator requires non-empty `scene_elements`, every target word anchored in some element's `vocab_terms` AND used in its caption, a non-empty `intro_text` that uses every target word, and rejects `word:`/`word —` glossary captions. Generation (`step_infographic.py`): 2 substeps per candidate (`ig_design` → `ig_cloze`, premise-free) then `_step_infographic_image` renders one poster/candidate. Engine: `restart_infographic_from_substep(...)`; artifacts under `…/pack_{id}_{slug}/infographic_cand_{i}/`. Prompts: `infographic_design.txt`, `infographic_cloze.txt`, `infographic_image.txt`. Step keys `ig_design`/`ig_cloze` in `LLMStepConfig` (seeded across all 3 sets by migration `0037`). Selection: `POST /api/infographics/{id}/select/` → `infographic_selection_service.select_infographic_candidate` (mirrors GN select). **`ClozeItem` now has BOTH `novel` and `infographic` nullable FKs**: active/promoted cloze = **both NULL**; staged = one of them set. **All active-cloze reads must filter `novel__isnull=True, infographic__isnull=True`** (instructional_service + both generation_views review endpoints + both selection services' promote-delete) — filtering only `novel__isnull=True` would leak staged infographic cloze to students. The active set is shared across content types (last published wins; cloze is medium-agnostic)
- **Pipeline content-type gating**: `GenerationJob.content_types` (JSON, default `['graphic_novel']`; empty/legacy = GN only) selects which formats a job generates. `PIPELINE_STEP_ORDER` appends `INFOGRAPHIC_DESIGN` + `INFOGRAPHIC_IMAGE` after the GN steps; `orchestrator._run_step` skips a content type's steps when not in `content_types` (via `_step_content_type` + `job_generates_content_type`). `TriggerGenerationView` accepts `content_types`; the generation wizard offers GN/infographic checkboxes. **Choice is per-assignment**: `StudentWordSetAssignment.content_type` (`graphic_novel`/`infographic`, default GN) is set by the teacher in `AssignSetForm` (radio) and persisted by `AssignmentService.assign_word_set(content_type=...)` (updates existing assignments too). `instructional_service.get_pack_data` serves the student's chosen type's selected content, **falling back** to the other published type then legacy stories if the chosen one isn't published. **Assignment is gated on published content**: a content type is only offered when at least one pack in the set has a candidate of that type with `is_selected=True` — a set with neither published cannot be assigned. `teacher_views._available_content_types_for_word_set(word_set)` computes this; the `assignments/` GET returns `available_content_types` and the `assign/` POST rejects (400) an unpublished/unavailable `content_type` (guards direct API calls too). `AssignSetForm` shows a blocking notice when neither is published, a read-only label when exactly one is, and both radios when both are. Student reader: `InfographicReader.jsx` (routed in `InstructionalFlow` on `story.type === 'infographic'`) — the poster fills the same widened student shell as the graphic novel reader (`[data-app="student"].app-shell:has(.infographic-reader)` → `max-width: min(1792px, 90vw)` in `graphic-novel-reader.css`; posters are landscape 16:9 so full width fits without a height cap), and each vocab word in the list below it carries a `TextToSpeechButton` (browser Web Speech `speechSynthesis`, same component the practice/primer views use — no backend audio). Admin review: `PackInfographics` in `GenerationReview.jsx`; review endpoints return `infographics` (array) + `infographic` (selected/null)
- The single GN generation engine is `restart_graphic_novel_from_substep(job, pack_id, substep_key, words_data, candidate_index)` — runs one candidate from a substep to the end and persists novel + staged cloze. `_step_graphic_novel_script` is pure orchestration: loops packs × candidates, skips complete candidates (novel-with-pages), resumes incomplete ones from the first substep lacking a COMPLETED log. Both fresh runs and resume delegate to the engine (removed the old ~200-line duplicate). Resume keys + COMPLETED logs + artifacts are all per-`(pack, candidate_index)`; artifacts live under `temp/generation_artifacts/job_{id}/pack_{id}_{slug}/cand_{i}/`. Substep restart API gains an optional `candidate_index` (defaults 0)
- Unified graphic novel length: each candidate runs at a length **derived deterministically from the pack's word count** — `page_count_for_word_count(n)` in `services/generation/constants.py` returns 5 for ≤4 words and 6 for >4 (threshold `GRAPHIC_NOVEL_WORD_COUNT_PAGE_THRESHOLD = 4`). `_step_graphic_novel_script` / `restart_graphic_novel_from_substep` compute `required_page_count`, pass it into the router/scorer/beat/final prompts (so premises are designed for that length), and then **force** `winning_premise['page_count'] = required_page_count` regardless of what the LLM returned. The router prompt is told the required length (every premise must use it); the scorer treats length as fixed and equal across premises. Beat-sheet and final-script substeps dispatch to the matching template via `GRAPHIC_NOVEL_BEAT_SHEET_TEMPLATES` / `GRAPHIC_NOVEL_SCRIPT_TEMPLATES`. The 5- and 6-page prompt files stay separate (6-page beat sheet adds a "deepen" note; `page_turn_question` is null for the final page)
- 6-call Gemini workflow per pack: team selector (coin-flips solo/dual via `sample_team_options()`, decides `vault_framing`) → router → scorer → cloze → beat sheet → final script, each with its own DB config key. Substeps use a system/user prompt split (instructions in system, data in user) — no `{input_json}` substitution. Each substep retries up to 3 attempts internally (`max_retries=2` on the configured primary site, no fallback site); the orchestrator does NOT retry the whole GRAPHIC_NOVEL_SCRIPT step
- Cloze generation is a dedicated substep (index 3) after scoring, using only the winning premise + vocab words (independent of beat sheet/final script). Prompt `graphic_novel_cloze.txt`, config key `gn_cloze_gen`, artifact `03b_cloze_generation.json`
- Per-pack resume: `_step_graphic_novel_script` resumes each pack from the first substep without a COMPLETED log (`_resume_substep_index_for_pack`), reusing prior on-disk artifacts — a job that failed mid-pack resumes at the failed substep, not team selection. A fresh pack runs the full workflow. The COMPLETED log is authoritative (artifacts write before validation, so a stale artifact can exist without one)
- Substep restart: `POST /api/generation-jobs/{id}/restart-substep/` (`pack_id` + `substep`) reconstructs context from prior artifacts and reruns from that substep, then runs the full script step over ALL packs before marking the job COMPLETED — filling packs the original run never reached. The skip-guard leaves complete packs untouched; a remaining-pack failure marks the job FAILED
- Canon files in `backend/data/canon/` load at runtime via `canon_service.py`, split by purpose: `team-selector-summaries.md` (team selection), `script-character-sheets.md` (narrative-only for router/scorer/beat sheet), `vault-summary-premises.md` (condensed vault for premises), full visual sheets collapsed via `collapse_markdown()` (final script + images). Per-character Lexi Mini examples ground vocab as a creature-summon mechanic, not generic magic
- World context preamble lives in `prompts/graphic_novel_world_context.txt`, injected at runtime by `_run_graphic_novel_substep` between the role line and step instructions — individual templates do NOT contain it
- Validators take `(result, ctx=None)` where `ctx` is a `SubstepContext` dataclass (`graphic_novel_validators.py`) accumulating `target_terms`/`winning_premise`/`selected_away_team`/`router_result`; build via `SubstepContext.from_input_summary(...)`. No lambda closures
- Image prompts: prose panel formatting, trimmed synopsis for pages 2+, per-character Ink-color highlighting — vocab words render in the character's Ink color (orange for Hugo) while surrounding caption text stays white/cream (never gold/yellow, for contrast)
- Review page: no story title; uses story characters; definitions from `PrimerCardContent.kid_friendly_definition` (fallback: `WordDefinition` truncated to 8 words)
- Script pacing: page 1 needs a WHO/WHERE/WHAT establishing panel; the final page (5 or 6) uses tension-before-resolution. Re-establishing caption on location change — when a page's location changes via a move the reader can't infer by simple walking (magical/instant or a far jump), the first panel opens with a short caption re-establishing WHERE and HOW the character arrived; ordinary walkable moves get none (saves the 80-word budget). Prompt decision test: "could a reader assume the character just walked there?"
- Pack grouping uses free-text `narrative_approach` (not a fixed engine list); downstream uses `central_thread` (not `central_problem`) to allow non-conflict narratives
- Prompt schemas use chain-of-thought field ordering: rationale/planning fields precede decision fields. Router `vocab_integration_plan` before the `premise` paragraph; scorer `dimension_rankings` scratchpad before `scores`; beat sheet `vocab_roles` + `ink_usage` before the `beat_sheet` array; final script per-page `page_planning` before `panels`; team_rationale before booleans
- Vocab semantic accuracy: target words must stay semantically true to the story's resolution — never false accusations, rumors, or red herrings that get debunked (ESL learners build meaning from outcomes)
- Lexi Mini system: writing a vocab word with a hero's tool (Hugo's carpenter's pencil, Leo's wax crayon, Amara's quill, Mei's marker) summons a temporary monochromatic creature that acts out the definition; tools only appear on-page during a summon. Hard cap 0–1 Minis/story (`GRAPHIC_NOVEL_MAX_MINI_SUMMONS_PER_STORY = 1` in `constants.py`), enforced via the `total_ink_activations_planned` counter declared before vocab arrays in router + beat sheet. Most vocab integrates through dialogue/narration/world logic; every summon succeeds (no failure states). `GRAPHIC_NOVEL_ALLOWED_INTEGRATION_MODES` uses `lexi_mini_summon`; JSON keys `uses_direct_ink`/`ink_usage`/`total_ink_activations_planned` kept for backwards compat
- Pedagogical anchors: every router `vocab_integration_plan` item carries `pedagogical_anchor = {anchor_type, anchor_sketch}` with `anchor_type` ∈ `GRAPHIC_NOVEL_PEDAGOGICAL_ANCHOR_TYPES` ({demonstrated_action, near_synonym, category_example, visible_referent}); final script propagates these into `vocab_anchors`. `_validate_vocab_integration_plan` rejects premises missing the anchor
- Plot complexity caps: each premise carries `complexity_budget = {locations, secondary_characters, problem_thread}`. Location cap scales by length via `max_locations_for_page_count()` (≤2 for 5-page, ≤3 for 6-page); ≤2 secondary characters. `_validate_beat_complexity` enforces `setting_keys` count and that every `characters_featured` name maps to `away_team ∪ secondary_characters` — but matching is **distinctive-name-token overlap, not exact string equality**: the LLM labels the same character inconsistently across substeps (`Mr. Vidal` in the beat sheet vs `groundskeeper Mr. Vidal` in the premise), so `_significant_name_tokens()` lowercases, strips punctuation, and drops role/title stopwords (`mr`, `groundskeeper`, `the`, `professor`, …) before comparing. A featured name passes if its significant tokens overlap any allowed label's; a genuinely new character (`Mr. Stranger`) is still rejected. Same surface-variation tolerance as cloze/lexile matching — never exact-string-match LLM-emitted names
- Scoring dimensions: `GRAPHIC_NOVEL_SCORING_DIMENSIONS = {narrative_clarity, visual_potential, vocabulary_integration, pedagogical_clarity, character_fit}`. All premises share the same `required_page_count`, so the scorer judges how well each uses that fixed length (length is not a differentiator); location ceiling is page-count-aware
- Graphic novel image generation retries once on failure then stops (does not skip to later pages)
- Admin per-page image editing: `POST /api/graphic-novel-pages/{id}/edit-image/` (`{prompt}`) re-renders one page via OpenAI `images.edit` using the displayed image as reference; result saved to `edited_image` (original `image` never overwritten) and auto-selected. `POST .../select-image/` (`{variant}`) flips `use_edited_image`. Both admin-only. The edit is **asynchronous**: the view validates + reads the reference bytes synchronously, sets the page `RUNNING`, then runs the slow image call in a daemon thread (`_run_page_image_edit`) so no gunicorn worker is held for the ~30-60s wait; it returns **202** (or **409** if an edit is already running). The frontend polls `GET .../image-status/` every 10s until `COMPLETED`/`FAILED`. `select-image/` stays synchronous (instant DB flip). Every read path uses the `display_image` property, so the choice propagates to students/review/status/continuity. UI: `GraphicNovelPageEditor.jsx` (click-to-zoom `ImageLightbox`)
- Admin per-page image **redraw**: `POST /api/graphic-novel-pages/{id}/redraw-image/` (no body) replays the page's *original generation* payload — the template-built prompt (`build_page_image_prompt(page)`) + the previous page as the continuity reference (`previous_page_reference_bytes(page)`, both in `graphic_novel_images.py`) — to get a fresh roll of the same prompt when an image came out bad. Distinct from edit (which uses *this* page's image + an admin instruction). Shares the async machinery: validates + builds the payload synchronously, sets `RUNNING`, runs `_run_page_image_redraw` in a daemon thread, returns **202** (or **409** if busy), frontend polls the same `image-status/`. Result saves to `edited_image` + auto-selects (`prompt_used` tagged `[REDRAW]`), so the original is preserved and reversible via the variant picker. `build_page_image_prompt` was extracted from `_step_graphic_novel_images` so the pipeline and redraw stay in lockstep. UI: a "Redraw" button beside "Edit image" in `GraphicNovelPageEditor.jsx` (shared `pollUntilDone` helper)

- LLM config sets: there are **3 `LLMConfigSet` profiles** (seeded by migration `0031`, named "Set 1/2/3"), each holding its own full copy of the `LLMStepConfig` rows (`unique_together = (config_set, step_key)`). - Pipeline dispatch: orchestrator builds a `[primary, primary, primary, fallback]` attempt list from `LLMStepConfig` (3 attempts on the primary site, then 1 on the fallback site). `_call_llm_with_config(site_config, system_prompt, user_prompt)` routes by `provider_type`. On a recoverable attempt failure it writes a transient `FAILED` log whose `output_data` carries a per-site-role `retry_message` + structured fields (`failed_attempt`, `failed_site_role`, `next_attempt`, `next_site_role`, `failed_model`, `next_model`) via `_build_retry_payload(plan, failed_idx)`; attempts count *within each site role*

Any step wanting prose must request a JSON object with named fields and format the dict itself; calling `.strip()`/string methods on the result raises
- Question generation batch size is 2 words per LLM call (`QUESTION_BATCH_SIZE` in `step_questions.py`; small to avoid proxy timeout on slow models). Idempotent per batch: questions commit as each batch succeeds (no wrapping transaction), and resume/retry reads the job's existing `Question.word_id` set and skips batches whose words are all generated (clearing only partial batches) — no reprocessing or duplicates. `questions_created` is recomputed as the job total; COMPLETED log records `batches_skipped`. `restart_pipeline_from_step` for QUESTION_GEN still deletes all questions first (intentional full regen)
- **Sentence-writing (productive, LLM-judged) questions** (`SENTENCE_WRITE_GEN` step, migrations `0038`/`0039`): the only question type NOT auto-graded by exact string match. Two variants per word — `SENTENCE_WRITE_GUIDED` (mastery L4, scenario + sentence starter) and `SENTENCE_WRITE_OPEN` (L5, light scenario, no starter) — from two independent prompts (`sentence_write_guided.txt`/`sentence_write_open.txt`), gated to L4/L5 via `QUESTION_TYPE_LEVEL`, skill tag `sentence_production`. Generated in `step_sentence_write.py` (batch `SENTENCE_WRITE_BATCH_SIZE = 10`, idempotent per word per variant, Lexile-aware). **Guided-only floor**: when content Lexile (`_content_lexile` = target×0.85) ≤ `SENTENCE_WRITE_GUIDED_ONLY_MAX_LEXILE = 600` (generation `constants.py`), the OPEN variant is skipped and the GUIDED question is attached to BOTH L4 and L5 `suitable_levels` — lower-proficiency readers never get unscaffolded open production but still practice productively at L5 (done in generation code, not a prompt skip: open is a separate LLM call and levels are code-assigned). Reuses the `Question` model: scenario→`question_text`, model sentence→`example_sentence`, rubric anchors (`intended_sense`/`acceptable_use_notes`/`sentence_starter`)→`options`, `correct_answers=[]`. `QuestionSerializer` strips `options` + `example_sentence` for these types and exposes a student-safe `sentence_write` payload (variant/starter/definition/max_revisions) so the judge anchors + model sentence never leak. **Answer time**: `services/sentence_evaluation_service.py` runs the judge (own LLM step key `sentence_judge`, injection-hardened, structured `correct`/`almost`/`incorrect` verdict + `error_type` + coaching `hint`). The revision loop is **server-tracked** (Guided 3 / Open 2 revisions): attempt history ({sentence, hint, verdict} per try) lives in the session under `SubmitAnswerView._SW_SESSION_KEY` (single key, resets when the question changes — like the typo-retry flag), so the cap and the fragility decision cannot be reset by a client withholding its history; any `prior_attempts` in the request body is ignored. Judged submits are also gated on `daily_question_limit` (pending misses record no `UserAnswer`, so without the gate the judge LLM could be called without bound). `SubmitAnswerView._handle_sentence_write` returns `sentence_write_pending` + hint (+ server-truth `attempts_used`/`revisions_left`) on a non-terminal miss (no scoring), and only calls `PracticeService.process_answer(productive_judgment=...)` at the terminal step (correct / give-up / revisions exhausted). Scoring reuses `_classify_response_quality` via 3 rules — `productive_correct` (first-try, +5 XP bonus), `productive_recovered` (fixed after a genuine incorrect → fragile), `productive_missed` (**softened**: −1 point not −2, no demotion, still fragile). Verdict persisted on `UserAnswer.judge_result` (JSON) for analytics. **LLM-down = skip**: a circuit breaker (3 consecutive failures → 5-min cache flag) makes `NextPracticeWordView` exclude sentence-write types (student gets a receptive question); a submit-time failure discards without penalty. Also excluded when the same word served one in the previous answer (no back-to-back). New-jobs-only — existing sets get these only on regeneration or a `SENTENCE_WRITE_GEN` restart-step. Full design: `docs/feature_plan/design-sentence-writing-questions.md`
- Deduplication uses cosine similarity ≥ 0.92 on Qwen3 embeddings
- Translation step uses `term`-based matching (not substring matching on source_text); prompt returns a `term` field per object for reliable join
- Primer `kid_friendly_definition` is a concise 3-8 word phrase (not a sentence); used on primer cards and as the review-page definition source. Syllable breaks must preserve the word's exact spelling — only middle dots (·) are inserted between syllables, never respelled phonetically (e.g. "sim·i·le", not "sim·i·lee"); single-syllable words output without dots. `_sanitize_syllable_text()` in `step_packs.py` enforces this on save: it strips separators (·, ., -) and, if the result no longer spells the term letter-for-letter (case-insensitive), falls back to the plain term so a phonetic respelling never reaches students
- Questions without `lexile_score` must be included in practice/dashboard queries (don't filter them out)

## Testing

- Backend: pytest + pytest-django + factory-boy
- Run `pytest` from `backend/` directory
- Frontend: no test framework yet
- **Test isolation — cache must be cleared between tests**: there is no `CACHES` setting, so Django uses a process-global `LocMemCache`. pytest-django rolls back the DB per test but NOT the cache, so cache-backed DB-derived state leaks across tests (notably `llm_config_service`, key `llm_step_configs_all`, 5-min TTL — one test seeding `model-3-primary` poisons a later test expecting the migration-seeded default). `backend/tests/conftest.py` has an autouse fixture that `cache.clear()`s before and after every test to prevent this. Keep it; any new cache-backed service is already covered. When a test "passes alone but fails in the suite," suspect a cross-test state leak (cache like this, or a module-level global) — verify the actual first failure with `pytest -x` before assuming flakiness or a DB deadlock.

## Code Conventions

- Backend follows Django/DRF patterns: models → serializers → views → services
- Services handle business logic; views are thin
- Frontend uses functional components with hooks
- CSS uses plain files with a 5-theme student system (no CSS-in-JS)
- Cloze blanks from LLM vary in underscore count — use regex matching, not exact string match
- Definition/example translation lookups are centralized in `vocabulary/utils.py` (`get_definition_translation` / `get_definition_translations`) — import the shared helper instead of adding a per-service ContentType+Translation lookup
